# Configuration

Tinker uses a single declarative configuration file called `tinker.toml` that serves as the contract between Tinker and your project. It is language-agnostic, minimal, and typically auto-generated by `tinker init`.

## `tinker.toml` — Full Spec

Below is the complete configuration reference with all available sections and options:

```toml
# ── Database ──────────────────────────────────────────────
[database]
# Connection string source:
#   "env:VAR_NAME"  — read from environment variable (supports .env files)
#   "postgres://…"  — direct DSN
source = "env:DATABASE_URL"
# Database type: postgres | mysql | sqlite3
type = "postgres"
# Override the driver name (optional, auto-detected from type)
driver = ""
# Migration directory (relative to project root, auto-detected if not set)
migrate_dir = "migrations"
# Seed directory (relative to project root, auto-detected if not set)
seed_dir = "seed"

# ── API ───────────────────────────────────────────────────
[api]
base_url  = "env:API_BASE_URL"
spec      = "openapi.yaml"      # OpenAPI/Swagger spec (optional)
auth      = "env:API_TOKEN"     # Auth token source
auth_type = "bearer"            # bearer | basic | api_key | raw

[api.headers]                    # Additional default headers
X-Custom-Header = "value"

# ── gRPC ──────────────────────────────────────────────────
[grpc]
addr       = "env:GRPC_ADDR"
proto_dir  = "./proto"
reflection = true

# ── Log ───────────────────────────────────────────────────
[log]
files = ["logs/app.log", "logs/error.log"]

# ── Plugin ────────────────────────────────────────────────
[plugin]
dir = "plugins"
enabled = true

# ── Custom Commands ───────────────────────────────────────
[commands]
migrate = "go run ./cmd/migrate"
seed    = "go run ./cmd/seed"
test    = "go test ./..."
lint    = "golangci-lint run"
dev     = "air"
build   = "go build -o bin/app ./cmd/app"

# ── Multi-Environment ─────────────────────────────────────
[envs.staging.database]
source = "env:STAGING_DATABASE_URL"

[envs.staging.api]
base_url = "env:STAGING_API_BASE_URL"
auth     = "env:STAGING_API_TOKEN"

[envs.production.database]
source = "env:PRODUCTION_DATABASE_URL"

[envs.production.api]
base_url = "env:PRODUCTION_API_BASE_URL"
auth     = "env:PRODUCTION_API_TOKEN"
```

### Section Reference

#### `[database]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `source` | string | Yes | Connection string source. Use `env:VAR_NAME` to read from environment, or a direct DSN like `postgres://user:pass@host:5432/db` |
| `type` | string | Yes | Database type: `postgres`, `mysql`, or `sqlite3` |
| `driver` | string | No | Override the auto-detected driver name. Usually unnecessary |
| `migrate_dir` | string | No | Path to migration files (relative to project root). Auto-detected if not set |
| `seed_dir` | string | No | Path to seed files (relative to project root). Auto-detected if not set |

#### `[api]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `base_url` | string | Yes | API base URL source. Supports `env:VAR_NAME` or direct URL |
| `spec` | string | No | Path to OpenAPI/Swagger spec file (YAML or JSON) |
| `auth` | string | No | Auth token source. Supports `env:VAR_NAME` or direct value |
| `auth_type` | string | No | Authentication type: `bearer`, `basic`, `api_key`, or `raw`. Defaults to `raw` (sets `Authorization` header directly) |
| `headers` | map | No | Additional default headers sent with every request |

#### `[grpc]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `addr` | string | Yes* | gRPC server address source (e.g., `env:GRPC_ADDR` or `localhost:50051`) |
| `proto_dir` | string | No | Path to `.proto` files directory |
| `reflection` | bool | No | Whether server reflection is enabled. Required for native `list` and `describe` commands |

*Either `addr` or `proto_dir` must be set.

#### `[log]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `files` | string[] | Yes | List of log file paths (relative to project root or absolute) |

#### `[plugin]`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `dir` | string | No | Directory containing script plugins (default: `plugins`) |
| `enabled` | bool | No | Whether the plugin system is active (default: `true`) |

#### `[commands]`

A flat map of command names to shell command strings. Commands are executed via your system shell, so pipes, redirects, and shell features work natively.

#### `[envs.*]`

Multi-environment configuration. Each key under `[envs]` is an environment name (e.g., `staging`, `production`), and the value is a set of partial overrides that are applied on top of the base configuration. Only non-nil fields in the override are applied; the base configuration is left unchanged for any fields you don't override.

## Environment Variable Resolution

Any configuration value starting with `env:` is resolved from the environment at runtime. Tinker automatically loads `.env` files from your project root, so you can keep sensitive values out of your configuration.

### How it works

1. Tinker reads `tinker.toml` and parses all values
2. It loads `.env`, `.env.local`, and `.env.example` files into an in-memory map (no process-level mutation)
3. For any value like `env:DATABASE_URL`, it looks up `DATABASE_URL` in the loaded env vars
4. The resolved value is used for the actual connection/configuration

### Example

```bash
# .env
DATABASE_URL=postgres://user:pass@localhost:5432/mydb
API_BASE_URL=http://localhost:8080
API_TOKEN=secret-token
GRPC_ADDR=localhost:50051
```

```toml
# tinker.toml
[database]
source = "env:DATABASE_URL"
type   = "postgres"

[api]
base_url  = "env:API_BASE_URL"
auth      = "env:API_TOKEN"
auth_type = "bearer"

[grpc]
addr       = "env:GRPC_ADDR"
reflection = true
```

### Auto-Detected Environment Variables

When you run `tinker init`, Tinker scans your `.env` files for known database configuration variables. The following variables are recognized automatically and their database type is inferred:

| Variable | Inferred Type |
|----------|--------------|
| `DATABASE_URL`, `DB_URL` | Auto (by URL prefix) |
| `POSTGRES_URL` | PostgreSQL |
| `MYSQL_URL` | MySQL |
| `MONGO_URL`, `MONGODB_URI` | MongoDB |
| `DB_PATH`, `SQLITE_PATH`, `SQLITE_DB` | SQLite |
| `DB_HOST`, `DB_CONNECTION` | Auto (by value) |

File paths ending in `.db`, `.sqlite`, or `.sqlite3` are automatically recognized as SQLite databases.

## Multi-Environment Configuration

Tinker supports running commands against different environments (development, staging, production) without modifying your `tinker.toml` file. Environment overrides are defined in the `[envs.*]` sections.

### Defining environments

```toml
# Base configuration (development by default)
[database]
source = "env:DATABASE_URL"
type   = "postgres"

[api]
base_url  = "env:API_BASE_URL"
auth      = "env:API_TOKEN"
auth_type = "bearer"

# Staging overrides
[envs.staging.database]
source = "env:STAGING_DATABASE_URL"

[envs.staging.api]
base_url = "env:STAGING_API_BASE_URL"
auth     = "env:STAGING_API_TOKEN"

# Production overrides
[envs.production.database]
source = "env:PRODUCTION_DATABASE_URL"

[envs.production.api]
base_url = "env:PRODUCTION_API_BASE_URL"
auth     = "env:PRODUCTION_API_TOKEN"
```

### Using environments

```bash
# Use staging configuration
tinker --env staging db ping
tinker --env staging db tables
tinker -e staging api GET /users

# Use production configuration
tinker --env production db count users

# List available environments
tinker env list

# Show configuration for a specific environment
tinker env show staging
```

### Override behavior

Environment overrides are **partial** — only the fields you specify are replaced. All other fields fall back to the base configuration. This means you only need to specify the values that differ between environments (typically `source`, `base_url`, and `auth`), while shared settings like `type`, `auth_type`, and `reflection` are inherited from the base.

The `--env` / `-e` flag is a persistent flag, meaning it can be placed before any subcommand:

```bash
tinker -e staging db tables
tinker --env production api GET /health
```

## Auto-Detection

When you run `tinker init`, Tinker's auto-detection engine scans your project directory for:

1. **Environment files** — `.env`, `.env.example`, `.env.local` (parsed for database URLs, API base URLs, gRPC addresses)
2. **OpenAPI specs** — `openapi.yaml`, `openapi.json`, `swagger.yaml`, `swagger.json`, and other common naming patterns
3. **Proto directories** — `proto/`, `protos/`, `api/proto/` directories containing `.proto` files
4. **SQLite databases** — files ending in `.db`, `.sqlite`, `.sqlite3` in common locations
5. **Docker Compose files** — `docker-compose.yml`, `docker-compose.yaml`, `compose.yml`, `compose.yaml` (plus environment-specific variants)
6. **Migration directories** — `migrations/`, `db/migrations/`, `backend/migrations/`, and other common patterns
7. **Seed directories** — `seed/`, `db/seed/`, `db/seeds/`, `fixtures/`, and other common patterns
8. **Log files** — `*.log` files and common log directories

The detection results are used to generate a `tinker.toml` tailored to your project. You can always edit the generated file manually after initialization.

## Config Validation

Tinker validates your configuration when loading it and reports any issues. Common problems include:

- Missing or empty `source` fields in the `[database]` section
- Unresolvable environment variables (e.g., `env:VAR_NAME` where `VAR_NAME` is not set)
- Empty `base_url` in the `[api]` section
- Both `addr` and `proto_dir` empty in the `[grpc]` section

Use `tinker config show` to view your resolved configuration and verify that all values are correct.
